# Cleanup Guide — Files to Erase Before Fresh Deployment

This document lists all files and directories that should be cleared before
deploying to a new user or creating a clean VM image. These files contain
session-specific data, logs, cached credentials, and personal history from
previous deployments that should **not** carry over to new users.

---

## 🔴 Critical — Must Delete (Contains Personal/Session Data)

These files contain data from previous users and **will directly affect** new
deployments if not removed.

### 1. Agent Memory
```
memory/phantom_memory.md
```
**Why:** Contains the agent's accumulated memory of past sessions, tasks,
sites visited, login states, and personal user data (e.g. HN usernames,
Gmail accounts, task history). A new user should start with a blank memory.

**Action:**
```bash
rm -f memory/phantom_memory.md
# Recreate as empty file so the agent can write to it
echo "# Phantom Memory\n\n## Session History\n" > memory/phantom_memory.md
```

---

### 2. Browser Profile Data
```
phantom/browser_data/
```
**Why:** Contains the persistent Chromium browser profile — cookies, login
sessions, cached credentials, browsing history, localStorage, and IndexedDB.
This includes any logged-in sessions (Gmail, Twitter/X, LinkedIn, HN, GitHub,
etc.) from the previous user.

**Action:**
```bash
rm -rf phantom/browser_data/
mkdir -p phantom/browser_data/
```

---

### 3. Browser Screenshots
```
phantom/screenshots/
```
**Why:** Contains step-by-step screenshots taken during previous automation
tasks. May contain sensitive information from the previous user's browsing
sessions.

**Action:**
```bash
rm -rf phantom/screenshots/
mkdir -p phantom/screenshots/
```

---

### 4. Psiphon Tunnel Data
```
phantom/psiphon_data/
```
**Why:** Contains Psiphon tunnel state, cached server lists, and connection
history tied to the previous session.

**Action:**
```bash
rm -rf phantom/psiphon_data/
mkdir -p phantom/psiphon_data/
```

---

## 🟠 Important — Should Delete (Contains Runtime Logs & Cache)

These files don't contain personal data but will cause confusion or errors
if carried over (stale logs, wrong timestamps, cached channel IDs, etc.).

### 5. Application Logs
```
logs/
```
**Why:** Contains orchestrator and agent logs from previous sessions with
timestamps, task history, and debug output. New users should start with
clean logs.

**Action:**
```bash
rm -rf logs/
mkdir -p logs/
```

---

### 6. Slack Channel Cache (S3 / Local)
```
/root/.agent_settings.json
```
**Why:** Contains the cached Slack workspace info, channel ID, agent config,
and bot token from the previous user's workspace. A new user will have a
different Slack workspace and tokens.

**Action:**
```bash
rm -f /root/.agent_settings.json
```
> **Note:** This will be recreated automatically when `stage2_start.sh` runs
> `slack_interface.py config --set-agent` and `--set-channel`.

---

### 7. Orchestrator Lock File
```
.orchestrator.lock
```
**Why:** If the previous session crashed without cleanup, this lock file may
prevent the orchestrator from starting.

**Action:**
```bash
rm -f .orchestrator.lock
```

---

### 8. Python Bytecode Cache
```
**/__pycache__/
**/*.pyc
**/*.pyo
```
**Why:** Compiled Python bytecode is environment-specific and may cause
import errors if the Python version or paths differ between sandboxes.

**Action:**
```bash
find . -type d -name __pycache__ -exec rm -rf {} + 2>/dev/null || true
find . -name "*.pyc" -o -name "*.pyo" | xargs rm -f 2>/dev/null || true
```

---

### 9. Generated Settings File
```
settings.json
```
**Why:** Auto-generated by the orchestrator from `/root/.claude/settings.json`.
Contains API keys and model config specific to the current sandbox. Will be
regenerated correctly on next orchestrator start.

**Action:**
```bash
rm -f settings.json
```

---

## 🟡 Optional — Consider Deleting (Task Artifacts)

These files are task outputs that may or may not be relevant to keep.

### 10. Reports Directory
```
reports/
```
**Why:** Contains test reports and task output files from previous automation
runs. Unlikely to be relevant to new users.

**Action:**
```bash
rm -rf reports/
mkdir -p reports/
```

These files are part of the application and must be kept:

|------|--------|
| `agent-docs/` | Policy documents — required by orchestrator |

## 🚀 One-Command Cleanup Script

Run this before creating a new image or handing off to a new user:

```bash
#!/usr/bin/env bash
# cleanup.sh — Reset Phantom to a clean state for new deployment
set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"

echo "🧹 Cleaning Phantom deployment artifacts..."

# Critical: personal/session data
rm -rf "$SCRIPT_DIR/phantom/browser_data/"   && mkdir -p "$SCRIPT_DIR/phantom/browser_data/"
rm -rf "$SCRIPT_DIR/phantom/screenshots/"    && mkdir -p "$SCRIPT_DIR/phantom/screenshots/"
rm -rf "$SCRIPT_DIR/phantom/psiphon_data/"   && mkdir -p "$SCRIPT_DIR/phantom/psiphon_data/"
rm -f  "$SCRIPT_DIR/memory/phantom_memory.md"
printf "# Phantom Memory\n\n## Session History\n" > "$SCRIPT_DIR/memory/phantom_memory.md"

# Important: logs and runtime cache
rm -rf "$SCRIPT_DIR/logs/"                   && mkdir -p "$SCRIPT_DIR/logs/"
rm -f  /root/.agent_settings.json
rm -f  "$SCRIPT_DIR/.orchestrator.lock"
rm -f  "$SCRIPT_DIR/settings.json"

# Optional: task artifacts
rm -rf "$SCRIPT_DIR/reports/"               && mkdir -p "$SCRIPT_DIR/reports/"

# Python bytecode
find "$SCRIPT_DIR" -type d -name __pycache__ -exec rm -rf {} + 2>/dev/null || true
find "$SCRIPT_DIR" -name "*.pyc" -o -name "*.pyo" | xargs rm -f 2>/dev/null || true

echo "✅ Cleanup complete — ready for fresh deployment!"
echo "   Next: run ./stage2_start.sh to start the app"
```

Save as `cleanup.sh`, make executable with `chmod +x cleanup.sh`, and run before each new deployment.
